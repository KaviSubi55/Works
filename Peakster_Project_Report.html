1. INTRODUCTION

1.1 Topic and Purpose

Peakster is a comprehensive ski resort booking platform designed to simplify the process of planning and booking winter sports vacations. The project addresses the common challenges travelers face when organizing ski trips, such as booking accommodations, ski passes, equipment rentals, and ski school lessons across multiple platforms.

The primary purpose of this project is to create a unified, user-friendly platform that consolidates all aspects of ski vacation planning into a single, streamlined experience. By integrating multi-language support (English and Swedish), the platform caters to both international tourists and local Scandinavian markets, making winter sports more accessible to a broader audience.

1.2 Definition of Success

Success for this project is defined by the following criteria:

  • Functional Completeness: A fully operational booking system supporting multiple service types (accommodations, ski passes, equipment rentals, ski school)
  • User Authentication: Secure user registration, login, and profile management system
  • Multi-language Support: Complete internationalization with seamless language switching
  • Real-time Cart Management: Persistent shopping cart with real-time updates across devices
  • Security Implementation: Row Level Security (RLS) policies ensuring data protection
  • Modern Tech Stack: Utilizing cutting-edge technologies (Next.js 16, React 19, TypeScript)
  • Responsive Design: Mobile-first approach ensuring accessibility across all devices
  • Code Quality: Type-safe, well-structured, and maintainable codebase

1.3 Project Results

The Peakster project successfully achieved all defined success criteria, resulting in a production-ready web application with the following accomplishments:

2.3 Database Architecture

The application uses a PostgreSQL database hosted on Supabase with the following schema:

Table Purpose Key Features
users User profiles and authentication data Username, email, timestamps, UUID primary key
cart_items Shopping cart persistence Product details, quantities, JSONB for flexible data, user_id foreign key
orders Transaction history and order tracking Order status, total amount, JSONB order details, timestamps

All tables implement Row Level Security (RLS) policies to ensure users can only access their own data. Automatic timestamp triggers maintain created_at and updated_at fields, and cascading deletes ensure referential integrity.

2.4 Potential Issues and Solutions

Issue 1: Multi-Language State Management

Challenge: Managing translation state across components without prop drilling

Solution: Implemented a custom LanguageContext provider with TypeScript support, storing translations in structured JSON files and providing a useLanguage hook for easy access throughout the application.

Issue 2: Cart Synchronization Across Devices

Challenge: Keeping cart state synchronized when users access the platform from multiple devices

Solution: Database-backed cart storage using Supabase, combined with TanStack Query's cache management and optimistic updates for instant UI feedback while ensuring data consistency.

Issue 3: Next.js 16 Authentication with SSR

Challenge: Implementing secure authentication in Next.js 16's App Router with server-side rendering

Solution: Utilized Supabase SSR package with middleware-based route protection, separate server and client Supabase clients, and server actions for secure authentication operations.

Issue 4: Type Safety Across Full Stack

Challenge: Maintaining type safety from database to frontend

Solution: Zod schemas for runtime validation matching TypeScript types, server action return types, and Supabase-generated database types for end-to-end type safety.

3.3 Challenges Encountered

Challenge 1: Learning Curve with Next.js 16 App Router

Issue: Next.js 16 introduced significant changes to the App Router, server components, and data fetching patterns that differed from previous versions and traditional React applications.

Impact: Initial development was slower as understanding the distinction between server and client components, server actions, and the new routing paradigm required substantial research and experimentation.

Solution: Thorough documentation study, building small proof-of-concept features, and establishing clear patterns for server/client component boundaries. Created a consistent architecture using server actions for mutations and React Query for client-side state management.

Challenge 2: Supabase Row Level Security Configuration

Issue: Implementing proper RLS policies that balanced security with functionality, particularly for the signup process where new users need to create profile entries.

Impact: Initial signup attempts failed due to restrictive RLS policies preventing profile creation.

Solution: Implemented a service role key for the signup server action to bypass RLS during user creation, while maintaining strict RLS for all other operations. This ensures secure data access while allowing necessary account creation functionality.

Challenge 3: Type Safety with Dynamic Cart Data

Issue: Cart items need to store different types of product details (accommodation has check-in/check-out dates, ski passes have validity periods, etc.) while maintaining type safety.

Impact: Risk of runtime errors and difficulty in maintaining consistent data structures.

Solution: Utilized JSONB database field type for flexible storage combined with Zod schemas for validation and TypeScript discriminated unions for type-safe access to product-specific details.

Challenge 4: Multi-Language Architecture

Issue: Designing a scalable internationalization system without relying on heavy external libraries.

Impact: Potential for poorly organized translations and difficult maintenance as the application grows.

Solution: Built a custom language context with structured JSON translation files, TypeScript support for type-safe key access, and a simple useLanguage hook pattern that can easily scale to additional languages.

4.2 Limitations Encountered

Payment Integration:

The current implementation does not include actual payment processing. Integration with payment gateways like Stripe or PayPal would be essential for a production deployment but was beyond the scope of this project phase.

Email Notifications:

While user authentication is implemented, automated email confirmations for bookings, password resets, and account verification are not yet integrated.

Admin Dashboard:

The application currently lacks an administrative interface for managing inventory, viewing bookings, and updating content. This would be necessary for real-world operation.

Real-time Availability:

Accommodation and service availability is static. A production system would need real-time inventory management to prevent overbooking.

Search and Filtering:

Advanced search functionality with filters for price range, amenities, and dates could be expanded beyond the current basic implementation.
